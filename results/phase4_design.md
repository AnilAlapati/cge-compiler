# Phase 4 Experiment Design: Beyond Context Window Reasoning

## Research Question

**Can architecture maps enable LLM reasoning on repositories too large to fit in raw context?**

Specifically: When a repository's raw source code exceeds the LLM's context window, can a compressed architecture map (generated by our Phase 2 pipeline) provide sufficient structural information for the LLM to accurately answer architectural questions about the codebase?

## Motivation

### The Context Window Bottleneck

Current LLMs have finite context windows that impose hard limits on how much source code can be provided for analysis:

| Model | Context Window | Approximate Token Limit |
|-------|---------------|------------------------|
| gpt-4o-mini | 128K tokens | ~128,000 |
| gpt-4o | 128K tokens | ~128,000 |
| Claude 3.5 Sonnet | 200K tokens | ~200,000 |

### Ghostfolio: Our Test Case

The Ghostfolio repository is a production NestJS monorepo for personal finance/portfolio tracking:

| Metric | Value |
|--------|-------|
| **Raw source tokens** | **336,111** |
| **Architecture map tokens** | **58,405** |
| **Compression ratio** | **5.8x** |
| **Token reduction** | **82.6%** |
| **Source files** | 1,044 nodes |
| **Dependencies** | 839 edges |

> [!IMPORTANT]
> Ghostfolio's raw source code (336K tokens) **exceeds** gpt-4o-mini's 128K context window by 2.6x. It cannot be processed raw. However, the architecture map (58K tokens) **fits comfortably** within context, using only ~45% of the window.

This creates the ideal experimental condition: a repository where **Raw-only evaluation is impossible** but **Architecture Map evaluation is feasible**.

## Methodology

### Evaluation Approach: Architecture Map Only (AMO)

Since the raw source code cannot fit in context, we evaluate using only the architecture map as the knowledge source.

#### Step 1: Question Design (Complete)

28 high-quality architectural questions have been created in `benchmarks_real/ghostfolio/ghostfolio_questions.json`, covering:

- **Service dependencies** (e.g., "What services are injected into PortfolioService?")
- **Data flow** (e.g., "What is the data flow from activity creation to portfolio display?")
- **Module organization** (e.g., "What modules are in the endpoints subdirectory?")
- **Authentication & authorization** (e.g., "How is authentication handled across the API?")
- **Background processing** (e.g., "How are cron jobs organized?")
- **Entity relationships** (e.g., "What are the key entity relationships?")
- **Cross-cutting concerns** (e.g., "What interceptors exist and what do they do?")

Each question has a verified `expected` answer grounded in the actual source code.

#### Step 2: Architecture Map Generation (Complete)

The Phase 2 architecture map generator has been run on Ghostfolio, producing:
- `GENERATED_ARCHITECTURE.md` (233KB, ~58K tokens)
- `architecture_graph.json` (1,044 nodes, 839 edges)

#### Step 3: LLM Evaluation

For each question:
1. Provide the architecture map as context
2. Ask the architectural question
3. Collect the LLM's response
4. Score the response against the expected answer

#### Step 4: Scoring Rubric

Each answer is scored on a 0–3 scale:

| Score | Label | Criteria |
|-------|-------|----------|
| 3 | Excellent | Correct, complete, includes specific service/module names |
| 2 | Good | Mostly correct, may miss some details |
| 1 | Partial | Some correct elements but significantly incomplete |
| 0 | Wrong | Incorrect or completely off-base |

Scoring dimensions:
- **Accuracy**: Are the stated facts correct?
- **Completeness**: Are all relevant services/modules/relationships mentioned?
- **Specificity**: Does the answer reference specific class/file names?

### Experimental Controls

| Condition | Description | Feasibility |
|-----------|-------------|-------------|
| Raw Only | Full source code in context | ❌ **Impossible** (336K > 128K limit) |
| Map Only | Architecture map in context | ✅ Feasible (58K < 128K) |
| Map + Selective Raw | Map + targeted file contents | ✅ Feasible (58K + selected files) |

> [!NOTE]
> The inability to run a Raw Only baseline is actually **the point** of this experiment. We are demonstrating that architecture maps unlock reasoning on repositories that are otherwise inaccessible to LLMs.

### Comparison Baselines

To establish meaningful baselines:

1. **Zero-shot baseline**: Ask the same questions with no context (just the question). This measures how much an LLM can infer from general knowledge of NestJS patterns.

2. **Cross-repo comparison**: Compare Ghostfolio AMO scores against Phase 3 results for smaller repos where both Raw and Map conditions were testable. This validates whether the architecture map quality scales.

3. **Map + Selective Raw**: For a subset of questions, supplement the map with targeted raw files (e.g., the specific service file). This tests whether the map provides sufficient guidance for targeted retrieval.

## Success Criteria

### Primary Success Criteria

| Metric | Target | Rationale |
|--------|--------|-----------|
| Map Only mean score | ≥ 2.0 / 3.0 | "Good" or better on average |
| Map Only accuracy | ≥ 65% | Majority of answers are substantially correct |
| Map > Zero-shot | Δ ≥ 1.0 | Map provides significant lift over no context |

### Secondary Success Criteria

| Metric | Target | Rationale |
|--------|--------|-----------|
| Service dependency accuracy | ≥ 70% | Map should capture which services depend on which |
| Module organization accuracy | ≥ 80% | Map should clearly show module structure |
| Data flow accuracy | ≥ 50% | Harder; requires multi-hop reasoning |

## Why This Matters

### 1. The "Beyond Context Window" Argument

This is the strongest argument for architecture maps as a preprocessing step:

```
Without map:  336K tokens → ❌ Cannot analyze (exceeds context)
With map:      58K tokens → ✅ Can analyze (fits in context)
```

Architecture maps don't just compress code — they **enable analysis that was previously impossible**.

### 2. Practical Impact

Most production codebases are large:
- A typical microservice: 20K–100K tokens ✅ (fits raw)
- A NestJS monorepo (Ghostfolio): 336K tokens ❌ (needs map)
- A large SaaS platform: 1M+ tokens ❌ (needs map)

Architecture maps scale the applicability of LLM code analysis from small projects to real-world production systems.

### 3. Research Contribution

This experiment establishes:
- **Empirical evidence** that compressed architectural representations preserve sufficient information for reasoning
- **A benchmark** (28 questions with ground truth) for evaluating architectural understanding of Ghostfolio
- **A methodology** for evaluating LLM capabilities on large codebases using architecture maps
