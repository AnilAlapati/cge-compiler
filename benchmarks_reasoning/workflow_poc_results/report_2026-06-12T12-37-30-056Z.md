# Authentication Flow in Ghostfolio

This application supports multiple authentication methods. Here's how each works:

## 1. **Anonymous Login (Access Token)**

**Flow:**
1. Client calls `POST /auth/anonymous` with `{ accessToken: string }` (or deprecated `GET /auth/anonymous/:accessToken`)
2. `AuthController.accessTokenLogin()` → `AuthService.validateAnonymousLogin()`
3. The service:
   - Hashes the access token using `UserService.createAccessToken()` with a salt from config
   - Queries the database for a user with matching `accessToken`
   - If found: generates JWT via `JwtService.sign({ id: user.id })`
   - If not found: throws error (returns 403 FORBIDDEN)
4. Client receives `{ authToken: string }` JWT

---

## 2. **OAuth Login (Google/OIDC)**

**Google Flow:**
1. Client initiates: `GET /auth/google`
2. `AuthController.googleLogin()` triggers `AuthGuard('google')` → redirects to Google
3. Google redirects back to `GET /auth/google/callback`
4. `GoogleStrategy.validate()` (in `google.strategy.ts`):
   - Extracts OAuth profile: `profile.id` and provider type (`Provider.GOOGLE`)
   - Calls `AuthService.validateOAuthLogin({ provider, thirdPartyId })`
5. `AuthService.validateOAuthLogin()`:
   - Queries user by `{ provider, thirdPartyId }`
   - If not found AND signup enabled: creates new user via `UserService.createUser()`
   - If not found AND signup disabled: throws "Sign up forbidden" error
   - Generates JWT: `JwtService.sign({ id: user.id })`
6. `GoogleStrategy` returns `{ jwt }` to callback handler
7. `AuthController.googleLoginCallback()` redirects client to: `{ROOT_URL}/{LANGUAGE}/auth/{jwt}`

**OIDC Flow:** Identical, using `AuthGuard('oidc')` instead

---

## 3. **WebAuthn (Passkey/FIDO2)**

**Registration:**
1. `POST /auth/webauthn/generate-registration-options` (requires `AuthGuard('jwt')` + `HasPermissionGuard`)
2. `WebAuthService.generateRegistrationOptions()`:
   - Generates challenge via `@simplewebauthn/server`
   - Stores challenge in user's `authChallenge` field
   - Returns registration options to client
3. Client performs webauthn registration locally
4. `POST /auth/webauthn/verify-attestation` (requires JWT auth):
   - `WebAuthService.verifyAttestation()` verifies the credential
   - Creates `AuthDevice` record with credential data
   - Returns device info

**Authentication:**
1. `POST /auth/webauthn/generate-authentication-options` with `{ deviceId }`
2. `WebAuthService.generateAuthenticationOptions()`:
   - Loads device
   - Generates challenge, stores in user's `authChallenge`
   - Returns authentication options
3. `POST /auth/webauthn/verify-authentication` with `{ deviceId, credential }`
4. `WebAuthService.verifyAuthentication()`:
   - Verifies credential against stored device data
   - Updates device counter
   - Returns JWT: `JwtService.sign({ id: user.id })`

---

## 4. **Request Authorization (After Login)**

All subsequent API requests use the JWT from login:

**JWT Strategy Flow** (`jwt.strategy.ts`):
1. Client sends: `Authorization: Bearer {jwt}`
2. `JwtStrategy.validate()` (Passport JWT strategy):
   - Extracts JWT via `ExtractJwt.fromAuthHeaderAsBearerToken()`
   - Verifies signature using `JWT_SECRET_KEY`
   - Decodes to get `{ id }`
3. Loads full user: `UserService.user({ id })`
4. If subscription enabled:
   - Checks if user has `INACTIVE` role → returns 429 TOO_MANY_REQUESTS
   - Records analytics (country, activity count, timestamp)
5. Sets default currency/language if missing
6. Returns user object attached to `request.user`

**API Key Alternative** (`api-key.strategy.ts`):
1. Client sends: `Api-Key: {apiKey}` header
2. `ApiKeyStrategy.validate()`:
   - Validates API key exists via `ApiKeyService.getUserByApiKey()`
   - Loads user
   - Same subscription/analytics checks as JWT
   - Returns user object

---

## 5. **Permission Checking**

For endpoints requiring specific permissions:
1. `@HasPermission('PERMISSION_NAME')` decorator sets metadata
2. `HasPermissionGuard.canActivate()`:
   - Gets required permission from decorator metadata
   - Calls `hasPermission(user.permissions, requiredPermission)`
   - Returns 403 FORBIDDEN if unauthorized

**Summary:** Login generates JWT → JWT sent with requests → `JwtStrategy` validates & loads user → guards check permissions → request proceeds with `request.user` populated.

```typescript
// Model after GoogleStrategy
@Injectable()
export class GitHubStrategy extends PassportStrategy(Strategy, 'github') {
  public constructor(
    private readonly authService: AuthService,
    configurationService: ConfigurationService
  ) {
    super({
      callbackURL: `${configurationService.get('ROOT_URL')}/api/auth/github/callback`,
      clientID: configurationService.get('GITHUB_CLIENT_ID'),
      clientSecret: configurationService.get('GITHUB_SECRET'),
      passReqToCallback: true,
      scope: ['user:email']
    });
  }

  public async validate(
    _request: any,
    _token: string,
    _refreshToken: string,
    profile: Profile,
    done: DoneCallback
  ) {
    try {
      const jwt = await this.authService.validateOAuthLogin({
        provider: Provider.GITHUB,  // Ensure Provider.GITHUB exists in @prisma/client
        thirdPartyId: profile.id
      });
      done(null, { jwt });
    } catch (error) {
      Logger.error(error, 'GitHubStrategy');
      done(error, false);
    }
  }
}
```

# Market Data Fetching and Update Strategy

## Data Fetching Architecture

Market data is fetched through the **DataProviderService** (`data-provider.service.ts`), which acts as a facade to multiple data provider implementations. Key methods include:

### Primary Fetching Methods

1. **`getQuotes()`** - Fetches current market prices
   - Implements caching via Redis with `redisCacheService`
   - Groups requests by data source for efficiency
   - Respects rate limits via `getMaxNumberOfSymbolsPerRequest()`
   - Filters out derived currencies (e.g., `USDUSD`) for non-premium users
   - Updates market data via `marketDataService.updateMany()` after fetching

2. **`getHistoricalRaw()`** - Fetches historical market data
   - Handles derived currencies by converting to root currencies
   - Special handling for `USDUSD` (returns fixed value of 100)
   - Calls individual data providers' `getHistorical()` methods

3. **`getHistorical()`** - Retrieves cached historical data from database
   - Queries `MarketData` table with granularity filtering
   - Supports monthly and daily granularities

## Background Jobs System

The application uses **Bull queues** for background job processing:

### 1. Data Gathering Queue (`DATA_GATHERING_QUEUE`)

**Processor**: `DataGatheringProcessor` with two job types:

#### A. Historical Market Data Gathering
- **Job Name**: `GATHER_HISTORICAL_MARKET_DATA_PROCESS_JOB_NAME`
- **Concurrency**: Configurable via `PROCESSOR_GATHER_HISTORICAL_MARKET_DATA_CONCURRENCY`
- **Processor Method**: `gatherHistoricalMarketData()`
- **Process**:
  - Fetches historical data from date to present
  - Fills gaps with last known market price
  - Uses `marketDataService.updateMany()` for incremental updates
  - Uses `marketDataService.replaceForSymbol()` for forced updates (atomic transaction)
  - Marks delisted assets as inactive

#### B. Asset Profile Gathering
- **Job Name**: `GATHER_ASSET_PROFILE_PROCESS_JOB_NAME`
- **Concurrency**: Configurable via `PROCESSOR_GATHER_ASSET_PROFILE_CONCURRENCY`
- **Processor Method**: `gatherAssetProfile()`
- **Process**:
  - Fetches asset metadata (name, currency, ISIN, FIGI, holdings, etc.)

### 2. Portfolio Snapshot Computation Queue

**Processor**: `PortfolioSnapshotProcessor`
- **Job Name**: `PORTFOLIO_SNAPSHOT_PROCESS_JOB_NAME`
- **Function**: Computes portfolio snapshots and caches results in Redis
- **TTL**: Set based on whether there are calculation errors

### 3. Statistics Gathering Queue

**Processor**: `StatisticsGatheringProcessor` (not shown but referenced)

# Portfolio Sharing Feature Analysis

Based on the codebase, here's how the portfolio sharing feature works:

## Overview

The portfolio sharing feature allows **user A to grant user B access to view their portfolio** through an `Access` model that establishes a relationship between the portfolio owner and the grantee.

## Key Components

### 1. **Access Service** (`access.service.ts`)

The `AccessService` provides CRUD operations for managing access grants:

```typescript
- access()           // Retrieve a single access record
- accesses()         // Retrieve multiple access records with filtering
- createAccess()     // Grant access to another user
- deleteAccess()     // Revoke access
- updateAccess()     // Modify existing access permissions
```

### 2. **Access Controller** (`access.controller.ts`)

The `AccessController` exposes three main endpoints:

#### **GET /access** - Retrieve All Granted Access
```typescript
getAllAccesses(): Promise<Access[]>
```
- Returns all access records where the authenticated user is the **portfolio owner** (`userId`)
- Includes the grantee user information
- Maps results to indicate either `PRIVATE` (specific user) or `PUBLIC` (no grantee)

#### **POST /access** - Grant Access
```typescript
createAccess(data: CreateAccessDto): Promise<AccessModel>
```
- **Permissions Required**: `permissions.createAccess`
- **Parameters**:
  - `alias`: Optional custom name for the shared portfolio
  - `granteeUserId`: ID of the user to grant access to (optional for public sharing)
  - `permissions`: Array of permissions to grant
  - `userId`: Automatically set to the authenticated user (portfolio owner)

**Key Restrictions**:
- Checks if `ENABLE_FEATURE_SUBSCRIPTION` is enabled and user has `Basic` subscription → **Forbidden**
- Fails gracefully with `BAD_REQUEST` on database errors

#### **DELETE /access** - Revoke Access
```typescript
deleteAccess(id: string): Promise<AccessModel>
```
- **Permissions Required**: `permissions.deleteAccess`
- **Validation**: Verifies the authenticated user owns the access record before deletion
- Throws `FORBIDDEN` if user tries to delete access they don't own

#### **PUT /access/:id** - Update Access
```typescript
updateAccess(id: string, data: UpdateAccessDto): Promise<AccessModel>
```
- **Permissions Required**: `permissions.updateAccess`
- **Restrictions**: Same as `createAccess` (Basic subscription check)
- **Validation**: Verifies ownership before allowing updates
- **Operations**:
  - Update `alias` and `permissions`
  - Connect to a different `granteeUser`
  - Disconnect grantee for public sharing

## Permission Guards

All endpoints are protected by:

2. **`HasPermissionGuard`** - Checks user has required permission from `@HasPermission()` decorator

The guard verifies:
- User exists in request context
- User has required permission in their `permissions` array
- Throws `FORBIDDEN` (HTTP 403) if permission missing

## Subscription Restrictions

When `ENABLE_FEATURE_SUBSCRIPTION` is enabled:
- **Basic tier** users **cannot**:
  - Create new access grants (POST)
  - Update existing access grants (PUT)
  - Access is implicitly restricted by permission removal in user service

- **Premium tier** users can share portfolios freely

## Portfolio Access Usage

The sharing mechanism is used in the **portfolio controller** (`portfolio.controller.ts`) via:

```typescript
@Headers(HEADER_KEY_IMPERSONATION.toLowerCase()) impersonationId: string
```

This allows accessing impersonated user portfolios after validating the `Access` relationship through the `ImpersonationService`.
